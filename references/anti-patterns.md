# 锚点机制（诊断参考）

**用法：** 选锚点凭判断即可。锚点感觉不对、产出与无锚点的普通输出无异、回写入库（maintain）、或无人审阅的执行（委派 / 自动）时，翻这里逐项核对——库是永久复用、无人每次审，所以入库时锚点检验是必过项。核对时给出具体推理，不机械打勾。

---

## 黄金原则

**锚点不注入知识，锚点激活已有知识。**

下面全部是这一条的推论：

- 模型里这个人不显著 → 激活不出来 → 锚点无效
- 判断维度是机械的 → 不需要锚点，用规则
- 价值取决于"召唤强度"，不是"听起来权威"

## 锚点检验

人 + facet 够不够格，逐项核对，全部通过才召唤得动（清单可增减，互不影响）：

- **训练留痕充分** —— 这人在该领域有充分训练留痕（书 / 库 / talks / 主流讨论），搜名字第一页是权威结果。否则是 vanity，激活不出内容。
- **facet 精准** —— 要"具体"，也要"有方向"：
    - **具体**：一句话说清这人在**该维度**的立场，不是空泛形容词（"清晰、易读"）。
    - **有方向**：同一个倾向有正用形态、也有退化形态，facet 必须写明朝哪边使，否则模型会默认渲染成退化的中位形态。例："诘问"不导向 → 退化成解构 / 抬杠，要写成"为进入问题、还原真实处境而问"。方向要写进 facet 本身——想靠在锚点后面挂规则补方向，就是 facet 没到位。
- **维度对口** —— 这人最著名的作品正是这个生态 / 这个维度。借名气跨生态（BurntSushi 用在 Java）、或用错维度（Bloch 谈测试），都会激活到错误区域；这种产出表面通顺、仅靠阅读未必看得出，必须在选锚点时挡掉。

任一不过 → 换人，或换 facet。

## 一条边界：客观约束不靠锚点

语言 / 工具 / 格式 / 平台陷阱（"用中文"、"Java 17+"、"transform scale 的坑"）不是判断、激活不出来 → 进 domain 的「必要规则」段。但不要把锚点 facet 已经覆盖的内容又写成规则——冗余时保留锚点。
