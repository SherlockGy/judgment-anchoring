# 执行流程

主代理的角色是 **prompt 组装器 + reviewer**,不是 patcher。
- 从 domain README 把锚点配方 + 必要规则**抽取**成一份 prompt
- indwell 这份 prompt 直接产出
- 产出后做直觉判断,不合格就重新组装 prompt 重做

Domain README 的两段结构("锚点配方表" + "必要规则")定义见 `SKILL.md`。执行时只以**组装后的 prompt**为操作框架,不让其它上下文回流稀释锚点(细则见 Step 2)。

## Step 0: 路由

从 `references/anchor-catalog.md` 开始,按任务特征匹配已验证的领域配方:

- **明确命中一个 domain** → 走 Step 1 · 路径 A(应用预设)
- **匹配到 ≥2 个 domain** → 报告所有候选让用户选择,再走路径 A
- **未命中任何 domain** → 走 Step 1 · 路径 B(现场创建)

**进入 Step 1 前必须明文公示匹配结果**,例如:

> Catalog 匹配结果:命中 `java-business` / 未命中 / 多候选 `[A, B]`

公示行不存在 = 违反。

## Step 1: 组装 prompt

### 路径 A:应用预设配方(命中)

打开 `references/domains/{命中领域}/README.md`。domain 里只有两段,都要进这份 prompt:

- **锚点配方表** — 从表里选 2-5 个最贴合本次任务的维度,每个锚点保留"维度 + 人名 + facet"完整内容
- **必要规则** — 按本次任务语境过滤掉不适用的(例如任务不涉及测试就跳过测试相关规则),但适用的必须全进

进入 Step 2。

### 路径 B:现场构造锚点(未命中)

通知用户:

> 当前任务在 anchor-catalog 中未匹配到已验证配方。我将按锚点构造规则现场创建一组锚点。
> 如果你想自己指定锚点,告诉我"用 X / Y / Z"。

现场做四件事:

1. **维度盘点** — 列出 2-5 个判断密集的独立维度(明文写出,不允许在心里默念)
2. **候选人提名** — 每个维度提名 1 个锚点
3. **必要规则构造** — 写出 1-5 条该领域的必要规则(语言 / 工具 / 输出格式 / 客观失败模式)
4. **锚点把关** — 判断每个锚点是否够格(训练留痕充分、facet 精准、维度对口、与其它锚点不重复);拿不准时翻 `references/anti-patterns.md` 查对应失败模式。**无人实时审阅的执行(委派 / 自动)才需要逐项核对锚点检验。**

找不到强候选时,本维度留空白并明文标注"无强锚点"。

通过后进入 Step 2。

## Step 2: 执行

执行这份 prompt:开头公示激活的锚点,然后只以这份 prompt 为操作框架直接产出——锚点的纯度是它起作用的前提。

**若 domain README 注明需要求证**(判断密集、结论依赖现实认识的 domain),产出判断后按 `references/verification.md` 做独立求证;普通产出型 domain 不需要,直接产出即可。

### prompt 的结构

这份 prompt 只有以下结构,顺序固定:

```
<任务一句话>

激活锚点:
- <维度> | <人名>《作品》| <facet>
- <维度> | <人名>《作品》| <facet>
- ...

必要规则:
- <规则 1>
- <规则 2>
- ...

开头公示你激活的锚点,然后直接产出。
```

就这些。任何其他内容都不塞:不塞 checklist,不塞 self-report 模板,不塞架构要求清单,不塞过去对话的原始历史。塞了就是规则清单反模式复发到产出里。

## Step 3: 看产出

读一遍:**活的,还是平的?**

- **活的** — 有你 prompt 里没明写、却自然涌现的角度 / 连接 / 判断 → 锚点激活了 → 交付。
- **平的** — 只是 prompt 的字面执行,和无锚点的中位输出没区别 → 锚点没激活 → **换锚点重来,不在已有产出上打补丁**:回 Step 1 重选锚点或改 facet。若同类问题反复冒出,多半缺一个维度的锚点 —— 补维度而非补规则。

默认你(或用户)就在现场逐轮看 —— 这一步靠眼睛即可,不必做成流程。**只有无人实时盯着的执行(委派 / 自动)才需要把'活 / 平'判断刻意写成自检,并在信心不足处标注'需人审'。**

## Step 4: 回写建议(仅路径 B,且产出质量高时)

**触发条件**(必须同时满足):

- 走的是 Step 1 · 路径 B
- 锚点经得起 `anti-patterns.md` 的诊断(无明显失败模式)
- 产出是"活的"(Step 3 判定锚点已激活)
- 没有需要人审的关键部分

满足条件 → 主动询问用户:

> 本次现场创建的配方在 [领域] 类型任务上似乎工作良好。是否要回写到
> `references/domains/`?回写后下次同类任务自动匹配。
>
> 如果同意,我将走 `flows/maintain.md` 的流程把它加进库。

**如果自评质量一般(任一条件不满足)→ 不询问,避免打扰用户。**

## 违反就停

任一发生立即停止:

- **未公示路由匹配结果**(含多候选时自动选第一个、不问用户)— 你无法在执行前纠偏
- **凭空生成不存在的人名** — vanity anchor,唯一会静默失败、光读产出也未必看得出的
- **prompt 结构超出"锚点 + 必要规则 + 任务描述"** — 规则清单反模式复发,污染锚点

其余(选错锚点、facet 不精、跨生态搬锚点、产出"平了"、反复同类失败……)不是硬停,是 `anti-patterns.md` 里的诊断条目 —— 产出"平了"时翻开查,换锚点重来即可。
